MARGoS

MARGoS is research infrastructure that runs, validates, compares, and shares Multi-Agent Reinforcement Learning (MARL) experiments built on ARGoS and Ray RLlib — through a single, convention-driven command-line interface.

A typical MARL workflow is fragmented across a simulator, an RL framework, ad-hoc training scripts, manual seeding, scattered log files, and hand-built archives for sharing results. MARGoS unifies that workflow behind five commands. You provide names; MARGoS resolves paths, freezes configuration, captures the environment, propagates seeds, logs metrics, and packages everything needed to reproduce a run on another machine.

What MARGoS does

Run

Orchestrated training

One command loads a config, freezes it, seeds every RNG, launches the simulator, and runs RLlib training with live metrics.

Reproduce

Deterministic by design

Centralized seeding, config hashing, integrity checks, and an environment fingerprint make runs verifiable.

Compare

Validate outcomes

Compare a run against a reference for handoff success and strict reproducibility within a tolerance.

Share

Portable bundles

Export a run to a single ZIP and import it elsewhere — config, logs, scenario, script, and checkpoints included.

The five commands

CommandAliasPurpose
margos runp rRun an experiment from a config file
margos showp sList configs, results, imported runs, and bundles
margos comparep cCompare two runs for reproducibility / handoff
margos exportp ePackage a run into a shareable bundle

Scope

MARGoS is the software platform. This site documents how the software is built and used. Research questions, hypotheses, and the evaluation protocol are intentionally out of scope here and are described in the accompanying thesis/paper.

Technology stack

  • CLI: Typer + Rich (tables, progress, colored output)
  • Config: Pydantic schema + PyYAML
  • RL: Ray RLlib (PPO) on PyTorch
  • Simulation: ARGoS 3 with custom C++ controller & loop-functions plugins
  • Bridge: ArgosToZoo — a PettingZoo ParallelEnv talking to ARGoS over ZeroMQ
  • Analysis: NumPy + Matplotlib; TensorBoard for live curves

Getting Started

Installation & Setup

MARGoS has a pure-Python core and an optional simulation layer. Install only the core to explore the CLI, or the full stack to run training against ARGoS.

1 · Install the Python package

# core CLI (Typer, Pydantic, PyYAML, Matplotlib, Rich)
pip install -e .

# add the ML / simulation stack (RLlib, PyTorch, PettingZoo, ZeroMQ, …)
pip install -e ".[ml]"

# add developer tooling (pytest, ruff, type stubs)
pip install -e ".[dev]"

This registers two console scripts: margos and its short alias p. Requires Python >= 3.10.

Dependency groups

GroupPackages
coretyper, pyyaml, pydantic, matplotlib, rich
mltorch, tensorboard, numpy, ray[rllib], pyzmq, pettingzoo, gymnasium
devpytest, ruff, types-PyYAML

2 · Build the ARGoS plugins

Running actual training requires ARGoS 3 and two compiled C++ plugins (a robot controller and a loop-functions module). Prerequisites: ARGoS 3 on the argos3 path, CMake ≥ 3.8, and the cppzmq library.

argos3 --version            # verify ARGoS is installed

cd argos_plugins
mkdir -p build && cd build
cmake ..
make -j4

This produces the two shared libraries MARGoS loads at runtime:

argos_plugins/build/controllers/libmy_ipc_controller.dylib
argos_plugins/build/loop_functions/libzoo_loop_functions.dylib
No manual path editing

You do not substitute library paths by hand. Scenario files ship with the placeholders MARGOS_CONTROLLER_LIB and MARGOS_LOOP_LIB; the training script calls prepare_scenario(), which writes a temporary scenario with the absolute plugin paths filled in. See ArgosToZoo Bridge.

3 · Verify the installation

margos --help                 # CLI help
margos show                   # list configs / results / bundles
pytest                        # fast unit tests (slow e2e excluded by default)
Platform note

The build paths and fingerprint use the macOS .dylib extension. On Linux the compiled artifacts are .so; adjust plugin paths accordingly for your platform.

Getting Started

Quickstart

MARGoS follows convention over configuration: you type names, MARGoS resolves the paths. Here is the full loop, from training to sharing.

Run an experiment

# resolves experiments/configs/aggregation_v1.yaml
margos run aggregation_v1
p r aggregation_v1            # short alias

# no name? pick interactively from configs + imported runs
margos run

Training launches the ARGoS simulator, runs RLlib PPO, streams a live progress bar, and (by default) starts TensorBoard at http://localhost:6006. When it finishes you get a summary table with final/best reward, AUC, duration, iterations, the config hash, and the output directory.

Inspect what you have

margos show                   # everything
margos show results           # just completed runs
p s configs                   # just configs

Compare two runs

margos compare aggregation_v1_20260101-101010 aggregation_v1_20260101-120000
p c                           # pick both runs interactively

You get a reproducibility table: SRQ5 handoff (reward agreement) and SRQ3 strict (reward + AUC + config hash + integrity), each within a ±1% tolerance.

Share a run

# results/<run>/  ->  bundles/<run>.zip
margos export aggregation_v1_20260101-101010

# on another machine
margos import bundles/aggregation_v1_20260101-101010.zip

Import unpacks into experiments/imported/<name>/, rewrites paths to the bundled scenario and training script, and prints an environment comparison so you know whether the run can be faithfully reproduced.

Tip

Every command accepts --help, and a global --verbose / -v flag prints full tracebacks when something fails.

Concepts & Architecture

Configuration

An experiment is described by a small YAML file that references a scenario and a training script. Hyperparameters live in the training script, not the config — the config only bundles references and the seed.

Schema

experiment:
  name: "aggregation_v1"      # required, non-empty
  seed: 42                    # required, integer >= 0

scenario:
  file: "scenarios/footbot_aggregation.argos"   # required

training:
  script: "training/aggregation.py"   # required
  iterations: 300             # default 100, integer >= 1
  tensorboard: true           # default true

output:
  dir: "results/"             # default "results/"

Fields

FieldTypeRule
experiment.namestrmin length 1
experiment.seedint≥ 0
scenario.filestrpath to a .argos file, resolved & checked for existence
training.scriptstrpath to a Python script exporting main()
training.iterationsint≥ 1, default 100
training.tensorboardbooldefault true
output.dirstrdefault "results/"

Path resolution

The scenario.file and training.script paths are resolved relative to the experiments/ directory (the parent of configs/). If either file does not exist, validation fails with a clear error before any training begins.

You typeMARGoS resolves
margos run exp_v1experiments/configs/exp_v1.yaml
scenario.file: scenarios/x.argosexperiments/scenarios/x.argos
training.script: training/x.pyexperiments/training/x.py

Validation & hashing API

from margos.config import load_config, hash_config, save_frozen_config

config = load_config("experiments/configs/aggregation_v1.yaml")  # -> MargosConfig
digest = hash_config(config)         # deterministic SHA-256 of the normalized config
save_frozen_config(config, out_dir)  # writes config.yaml into the run directory

hash_config serializes the validated config to JSON with sorted keys and compact separators before hashing, so the digest is independent of field order or formatting. This hash is the backbone of reproducibility checks.

Why hyperparameters stay in scripts

Keeping learning rates, batch sizes, and network shapes inside the training script means the config stays a tiny, stable reference. The script is itself captured in the run directory and in export bundles, so nothing is lost — but the config you edit by hand never balloons.

Concepts & Architecture

Reproducibility

Reproducibility is the central concern of MARGoS. Four independent mechanisms work together so a run can be re-executed and verified — not just re-launched.

Layer 1

Seed propagation

Every RNG is seeded from one value, before the training script is imported.

Layer 2

Config hashing

The frozen config is hashed (SHA-256) for identity comparison.

Layer 3

Integrity check

The hash is recomputed after training to prove the config never changed.

Layer 4

Environment fingerprint

Python, OS, package, runtime, and build versions are captured.

1 · Centralized seeding R3.1

The orchestrator calls set_all_seeds(seed) before dynamically importing the training script, so any module-level RNG initialization is already deterministic. It seeds:

random.seed(seed)
numpy.random.seed(seed)
torch.manual_seed(seed)
torch.cuda.manual_seed_all(seed)   # when CUDA is available

ARGoS is seeded separately: ArgosEnv.reset(seed=…) rewrites the scenario's <experiment random_seed> attribute and restarts the simulator, because ARGoS fixes its RNG at experiment initialization. Robot start positions and orientations are then drawn from ARGoS's seeded "argos" RNG, making placement deterministic for a given seed.

2 · Config hashing R3.3

On every run, the validated config is frozen to config.yaml and its SHA-256 written to config_hash.txt. Two runs with identical configs produce identical hashes.

3 · Integrity verification

After training completes, the orchestrator re-reads and re-validates the frozen config, recomputes the hash, and writes config_integrity.yaml:

start_hash: 3f8a…           # hash written at run start
end_hash:   3f8a…           # hash recomputed after training
match:      true
source:     config.yaml

If the frozen config changed during the run, the run fails loudly — the result cannot be trusted.

4 · Environment fingerprint

capture_fingerprint() records the surrounding environment into env_fingerprint.yaml:

SectionCaptured
python / os / platformInterpreter version, OS string, full platform
packagesray, torch, numpy, pydantic, gymnasium, pettingzoo, pyzmq, tensorboard
runtimemachine, processor, GPU availability, ARGoS version
buildgit commit + SHA-256 of each compiled ARGoS plugin

The fingerprint travels inside export bundles, so an importer can see exactly how far their environment diverges from the original. See Export & Import.

Verified, not assumed

Together these layers let margos compare report not only “did we get the same reward?” but “is this the same experiment, run unchanged, in a compatible environment?”

Concepts & Architecture

Error Handling

Every user-facing error follows one structured format (decision L8): what went wrong, the relevant context, and an actionable fix.

Format

Error: <what went wrong>
  <Key>: <value>
  <Key>: <value>
  Fix: <actionable suggestion>

Example

Error: Config file not found
  Path: experiments/configs/nonexistent.yaml
  Fix: Create the config file or check the experiment name

Error types

All errors derive from MargosError (message + context dict + fix). The CLI catches them and renders the format above; --verbose / -v additionally prints the full traceback.

ExceptionRaised when
ConfigNotFoundErrorA config file path does not exist
ValidationErrorConfig fails schema validation or a referenced file is missing
ExperimentNotFoundErrorA results / imported directory cannot be found
BundleNotFoundErrorA bundle file cannot be found
TrainingErrorThe training script is missing, malformed, or raises at runtime
ReportError / ComparisonErrorAnalysis fails (missing or empty metrics logs)
BundleError / ImportErrorExport or import validation fails
Design intent

The fix line is mandatory. An error that tells you only what broke — but not what to do — is treated as incomplete.

CLI & Workflow

Experiment Lifecycle

A single call to run_experiment(config_path) drives the whole pipeline. Each step writes its own artifact into a fresh, timestamped run directory.

  1. Load & validate config

    Parse YAML, validate against the Pydantic schema, and resolve the scenario & script paths relative to experiments/. Missing files fail here.

  2. Create the output directory

    results/<name>_YYYYMMDD-HHMMSS/ with logs/ and checkpoints/ subfolders.

  3. Freeze config + write hash

    Copy the validated config to config.yaml and write its SHA-256 to config_hash.txt.

  4. Capture environment fingerprint

    Record Python / OS / packages / runtime / build into env_fingerprint.yaml.

  5. Seed everything

    set_all_seeds(seed) runs before the training script is imported.

  6. Wire up logging

    Attach the JSONL MetricsLogger and, if enabled, the TensorBoardLogger; auto-launch TensorBoard on port 6006.

  7. Execute the training script

    Dynamically import the script and call main(config, callbacks, output_dir, progress=…).

  8. Verify integrity

    Recompute the config hash and write config_integrity.yaml; raise if it changed.

Output directory structure

results/<name>_YYYYMMDD-HHMMSS/
├── config.yaml             # frozen copy of the validated config
├── config_hash.txt         # SHA-256 of the frozen config
├── config_integrity.yaml   # start vs end hash + match flag
├── env_fingerprint.yaml    # Python / OS / packages / runtime / build
├── logs/
│   └── metrics.jsonl       # one JSON object per training iteration
├── tensorboard/            # TensorBoard event files (if enabled)
├── checkpoints/
│   └── final/              # RLlib checkpoint
└── report/                 # created by the analysis system
    ├── learning_curve.png
    └── summary.txt

Run summary

When margos run finishes it prints a Rich table summarizing the run: final reward, best reward, AUC, duration, iteration count, the (truncated) config hash, and the output path.

Return value

run_experiment returns (output_dir, tensorboard_process). The CLI owns the TensorBoard process lifetime and terminates it on exit.

CLI & Workflow

Comparison & Validation

compare_runs(run_dir, reference_dir, tolerance=0.01, reward_window=50) evaluates two runs along two distinct success criteria.

Two verdicts

VerdictPasses whenPurpose
SRQ5 handoffTail reward mean agrees within tolerance“Can someone else reproduce my result?”
SRQ3 strictReward and AUC and config hash and integrity all match“Is this the same experiment, bit-for-bit?”

Metrics compared

  • Tail reward mean — mean of the last reward_window (default 50) logged episode_reward_mean values, or all values if fewer were logged.
  • AUC — area under the reward curve via trapezoidal integration over iterations.
  • Config hash — equality of the two SHA-256 config digests.
  • Config integrity — both runs must report a clean start-vs-end hash match.

Each numeric metric is compared as a relative deviation against the reference, and counts as a match when the deviation is within tolerance (default ±1%).

Result table

The CLI renders a Rich table with both verdicts in the title and one row per metric — run value, reference value, deviation (and tolerance), and a Yes/No match. Zero-valued references are handled gracefully (a non-zero run against a zero reference yields infinite deviation rather than a crash).

Batch reproducibility

experiments/run_srq3_batch.py re-runs an experiment N times against a reference, appends per-run pass/fail rows to a CSV, and reports a success rate and reward standard deviation — the data-collection harness behind the reproducibility study.

CLI & Workflow

Export & Import

A bundle is a self-contained ZIP that lets another machine reproduce a run. Export packs it; import unpacks it and reports how compatible the new environment is.

Bundle contents

<experiment>.zip
├── manifest.yaml           # version, experiment name, export time, margos version
├── config.yaml             # frozen config            (required)
├── config_hash.txt         # config SHA-256
├── config_integrity.yaml   # integrity record
├── env_fingerprint.yaml    # environment fingerprint  (required)
├── logs/metrics.jsonl      # training metrics         (required)
├── scenario.argos          # the ARGoS scenario used
├── train.py                # the training script used
└── checkpoints/…           # RLlib checkpoints

Export refuses to build a bundle if any of the three required files is missing — a guard against shipping incomplete runs.

Export flow

  1. Validate

    Confirm the run contains config.yaml, env_fingerprint.yaml, and logs/metrics.jsonl.

  2. Write the manifest

    Record bundle version, experiment name, export timestamp, and MARGoS version.

  3. Pack artifacts

    Add config + hashes, fingerprint, logs, the scenario and training script, and checkpoints into the ZIP.

Import flow

  1. Validate structure

    The ZIP must contain manifest.yaml, config.yaml, and logs/metrics.jsonl, or import is rejected.

  2. Extract

    Unpack into experiments/imported/<experiment_name>/ (read from the manifest).

  3. Rewrite paths

    Point scenario.file and training.script at the bundled scenario.* and train.py so the run is immediately runnable.

  4. Compare environments

    Diff the bundled fingerprint against the local one and print the result.

Environment comparison

Import classifies differences into two tiers:

ResultMeaning
All matchEvery tracked component is identical.
Non-critical differencesSomething differs, but all reproduction-critical components match — a warning, not a block.
Critical differencesA critical component differs: ray, torch, numpy, gymnasium, pettingzoo, pyzmq, tensorboard, the ARGoS version, or a compiled plugin hash.
Permissive by design

Import and reproduction are never blocked by a mismatch — MARGoS warns and proceeds. The point is to make divergence visible, so you can judge how much weight a reproduced result carries.

Simulation Layer

ArgosToZoo Bridge

ArgosToZoo wraps the ARGoS C++ simulator as a PettingZoo ParallelEnv, so RLlib can train against physical-robot simulation. Python and ARGoS communicate over ZeroMQ with a small JSON protocol.

The environment

ArgosEnv launches argos3 as a subprocess and exposes a standard multi-agent RL interface. Agents are discovered at reset (robot_0, robot_1, …).

SpaceDefinition
ObservationDict{ position: Box(3), proximity: Box(24) in [0,1] }
ActionDiscrete(5): stop · forward · backward · turn left · turn right

Discrete actions map to controller commands (stop, forward_speed, backward_speed, left_speed, right_speed), which the C++ controller translates into differential-steering wheel velocities. Episodes truncate at max_steps.

Communication protocol

The ZMQ client is resilient: each request polls with a timeout and, on failure, recreates the socket and performs a ping handshake before retrying. Ports are chosen dynamically (an OS-assigned free port) and passed to the simulator via environment variables and/or injected into a temporary scenario file, so multiple environments can run concurrently.

The C++ plugins

PluginRole
my_ipc_controllerPer-robot controller: reads the 24-value proximity sensor, applies the current action to the differential-steering actuator.
zoo_loop_functionsOwns the ZMQ REP server, discovers controllers, randomizes start positions deterministically, applies actions in PreStep, and gathers observations in PostStep.
Separation of concerns

The C++ side is deliberately task-agnostic: it only ships raw sensor and position data. All MARL logic — rewards, metrics, termination — lives in Python. This keeps the simulator reusable across tasks.

Reward functions & scenarios

A task is defined by a single reward_fn callback with the signature (data) -> (team_reward, per_agent | None, metrics). The bundled aggregation_reward rewards robots for converging toward their shared centroid using distance, alignment, turning, and a perpendicular-movement penalty.

from margos.argos_zoo import ArgosEnv, aggregation_reward, prepare_scenario

scenario = prepare_scenario("experiments/scenarios/footbot_aggregation.argos")
env = ArgosEnv(argos_file=scenario, max_steps=100,
               reward_fn=aggregation_reward, seed=42)
obs, infos = env.reset(seed=42)

Scenario files are standard ARGoS XML describing the arena, walls, foot-bot distribution, physics, and the controller/loop-functions libraries (via the MARGOS_*_LIB placeholders that prepare_scenario fills in).

Simulation Layer

Training Scripts

Training scripts are ordinary Python files that the orchestrator imports dynamically. The only contract is a main() function with a fixed signature.

The contract

def main(config: dict, callbacks: list, output_dir: str, progress=None):
    """
    config     full experiment config (MargosConfig as a dict)
    callbacks  RLlib callbacks, including MARGoS' MetricsLogger
    output_dir directory for checkpoints and logs
    progress   optional TrainingProgress for the live progress bar
    """
    ...

The orchestrator inspects the signature and passes progress only if the script accepts it. If main() is missing, the script has a syntax error, or it raises at runtime, a structured TrainingError is shown.

Reference: the aggregation script

  • Calls prepare_scenario() to inject plugin paths into the scenario.
  • Registers an env creator that builds an ArgosEnv wrapped as a ParallelPettingZooEnv, seeded from the config.
  • Builds a PPO algorithm (PyTorch, old API stack), seeded via .debugging(seed=…), with an entropy bonus and a fixed batch configuration.
  • Runs the training loop, invoking each MARGoS callback's on_train_result manually (which avoids Ray serialization issues), and reports reward to the progress bar.
  • Saves the final checkpoint to <output_dir>/checkpoints/final.
Why manual callbacks?

MARGoS' loggers are called directly in the loop rather than registered with RLlib. This keeps the callback objects out of Ray's worker serialization path while still recording every iteration.

Quiet Ray output

setup_ray_environment() (called before importing Ray) and init_ray() suppress Ray's deprecation noise and configure minimal logging, keeping the CLI output focused on the progress bar and final summary.

Simulation Layer

Logging & Metrics

Metrics are written two ways at once: a durable JSONL file for analysis, and TensorBoard scalars for live monitoring. Both are RLlib callbacks attached by the orchestrator.

Per-iteration metrics

MetricsLogger appends one JSON object per training iteration to logs/metrics.jsonl:

{"iteration": 1, "timestamp": "2026-01-01T10:10:11",
 "episode_reward_mean": -2.13, "episode_reward_min": -9.8, "episode_reward_max": 4.1}
FieldMeaning
iterationTraining iteration number
timestampISO-8601 time the iteration finished
episode_reward_meanMean episode reward
episode_reward_min / maxReward spread across episodes

Metric extraction tolerates RLlib's varying result layouts, checking the top level, env_runners, and sampler_results for reward keys.

TensorBoard

When training.tensorboard is enabled, TensorBoardLogger writes reward/mean, reward/min, and reward/max scalars under <run>/tensorboard/, and the orchestrator launches a TensorBoard server:

tensorboard --logdir results/<run>/tensorboard   # auto-started on http://localhost:6006

The orchestrator frees port 6006 before launching and tears the server down when the CLI exits. If tensorboard is not installed, training proceeds and only the JSONL log is written.

JSONL vs TensorBoard

The JSONL log is the source of truth for reports, AUC, and comparisons; TensorBoard is for watching a run as it happens. They record the same metrics.

Testing

The suite uses pytest. Fast unit tests run by default; slow end-to-end tests that require ARGoS and RLlib are opt-in.

Running tests

pytest                    # fast tests (slow marker excluded via pyproject)
pytest -m slow            # only the slow end-to-end tests
pytest tests/test_config.py -v

The slow marker is configured in pyproject.toml; by default -m 'not slow' is applied, so a plain pytest skips the heavy simulation/training tests.
